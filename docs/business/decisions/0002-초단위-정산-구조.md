# ADR 0002 — 초 단위 트래픽 정산 구조

## 상태
Accepted

## 일자
프로젝트 착수 시

## 맥락

IPMax의 핵심 차별화는 **사용량에 정확히 비례하는 과금**이다.
- Consumer: 크레딧이 초 단위로 차감되어 "쓴 만큼만" 지불
- Provider: 자신의 회선으로 흐른 트래픽이 즉시 수익으로 집계
- 동시에 서버 부하·DB 트랜잭션이 폭증하지 않아야 함

## 결정

**라우터에서 N초 주기의 Heartbeat로 집계 값을 전송 → Kafka 토픽으로 스트리밍 → Redis에서 실시간 카운터·잔액 반영 → 일정 주기로 PostgreSQL에 원장 기록** 하는 2단 구조를 채택한다.

- **실시간(ms~초)**: Redis 카운터로 잔액·사용량 반영, 잔액 소진 시 즉시 세션 종료 신호
- **원장(분~일)**: PostgreSQL에 확정 트랜잭션 기록, 회계/정산의 단일 진실 원본
- **분석(일)**: ELK로 장기 패턴·이상 탐지

## 근거

- 완전 실시간(매 초 DB 커밋)은 비용·복잡도 대비 이득이 낮음
- Redis 단독은 내구성이 약해 회계 원장으로 부적합
- Kafka를 중간에 둠으로써: 버퍼링·재처리·장애 격리 가능

## 결과

- 긍정적
  - 사용자 체감 **즉시 반영**(Redis 잔액)
  - 대규모 세션 확장성 확보
  - 장애 시 Kafka offset으로 복구 가능
- 트레이드오프
  - **단기(수 초) 불일치** 가능: Redis 카운터와 최종 원장 간 미세 차이 → 세션 종료·일괄 확정 시 조정 필요
  - 운영 스택 증가 (Kafka·Redis 둘 다 HA 구성 필수)
- 후속 조치
  - "세션 종료 시 확정 트랜잭션" 알고리즘 상세 설계 (별도 ADR)
  - Heartbeat 주기·누락 허용치 수치 튜닝 ADR

## 대안 (Alternatives considered)

- **매 초 DB 커밋** — 단순하나 DB 부하·비용 과다
- **전량 Redis 기반** — 내구성 부족, 감사·회계 부적합
- **일 1회 배치 집계** — 실시간 잔액 반영 불가, Consumer 경험 악화

## 참고 자료

- Kafka 내구성 설정 가이드
- Redis persistence 문서
